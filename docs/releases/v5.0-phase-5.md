# Release Notes — v5.0-phase-5

## Release Overview

Phase 5 introduced Helm-based Kubernetes packaging, scheduled ETL execution, operational automation, and deployment lifecycle improvements.

The ETL platform evolved from raw Kubernetes manifests into a reusable, environment-aware deployment package with improved operational maturity.

## Major Features

## Helm Chart Packaging

Added a complete Helm chart structure for:

- PostgreSQL deployment
- ETL Job execution
- ETL CronJob execution
- Services
- ConfigMaps
- Secrets
- PersistentVolumeClaims

This enables:

- reusable deployments
- environment portability
- simplified operational workflows

## Kubernetes CronJob Support

Added scheduled ETL execution using Kubernetes CronJobs.

Features include:

- recurring ingestion scheduling
- cluster-managed execution
- autonomous ETL runtime orchestration

The platform now supports:

- manual Job execution
- scheduled CronJob execution

## Environment-Specific Deployment Profiles

Added:

- `values-dev.yaml`
- `values-prod.yaml`

This introduced:

- deployment profile separation
- reusable runtime configuration
- operational flexibility

## Resource Requests and Limits

Added Kubernetes resource governance for:

- PostgreSQL
- ETL Jobs
- ETL CronJobs

This improved:

- scheduler awareness
- runtime stability
- cluster resource management

## Helm Operational Automation

Added:

- Helm Makefile automation
- Helm deployment runbooks
- Helm NOTES.txt guidance
- Helm lifecycle commands

Supported operations include:

- install
- upgrade
- rollback
- uninstall
- release history inspection

## CI/CD Expansion

Expanded GitHub Actions validation to include:

- Helm linting
- Helm template rendering
- Kubernetes packaging validation

This improved:

- deployment confidence
- release validation
- packaging integrity

## Major Operational Improvements

### PostgreSQL Runtime Coordination

Implemented:

- initContainer startup validation
- `pg_isready` dependency checks
- Kubernetes-native runtime sequencing

This improved:

- deployment reliability
- workload startup coordination

### Kubernetes Deployment Validation

Added validation workflows for:

- Helm rendering
- runtime verification
- resource inspection
- database verification
- operational troubleshooting

## Known Limitations

Current platform limitations include:

- local Docker registry dependency
- single-node Kubernetes validation
- no centralized logging stack
- no Prometheus metrics integration
- no Grafana dashboards
- no distributed tracing
- no production ingress controller

These areas are planned for future phases.

## Upgrade Notes

Recommended validation commands after deployment:

```bash
helm lint helm/weather-etl
helm template weather-etl helm/weather-etl

helm upgrade --install weather-etl helm/weather-etl \
  --namespace data-pipelines \
  --create-namespace \
  --wait

kubectl get all -n data-pipelines

helm status weather-etl -n data-pipelines
```
